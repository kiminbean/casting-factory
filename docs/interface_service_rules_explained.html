0. 이 문서를 읽기 전에

이 문서는 casting-factory 프로젝트의 Interface Service 가 왜 지금처럼 설계되었는지 를 설명합니다. 단순히 "규칙 지켜라" 가 아니라, 각 규칙이 해결하려는 문제 와 깨지면 어떤 일이 일어나는지 를 함께 다룹니다.

읽는 대상

  • Python/FastAPI 는 쓸 줄 알지만 gRPC · 마이크로서비스 분리 는 해본 적 없는 주니어
  • 모놀리식 REST 는 해봤지만 실시간 스트림 vs polling 선택에서 막혔던 미드
  • 새 엔드포인트를 어디(Interface vs Management) 에 두어야 할지 망설여지는 팀원

이 문서를 읽고 나면

  • "이 기능은 backend/app/ 에 넣어야 할까, backend/management/ 에 넣어야 할까?" 를 5초 안에 판단
  • 코드 리뷰에서 "C-3 위반입니다" 같은 커뮤니케이션이 가능
  • 장애 발생 시 "어느 계층 문제지?" 로 범위를 좁혀 10분 안에 격리

읽는 데 걸리는 시간

처음 한 번: 약 30–45분. 이후는 참조용으로 필요한 섹션만 점프.

① 용어 사전 (먼저 익히면 편한 단어들)

Interface Service
웹 브라우저(Admin 포털, Customer 포털) 로부터 HTTP REST 요청을 받는 FastAPI 프로세스. 위치: backend/app/, 포트: 8000.
Management Service
공장 설비(로봇·컨베이어·AMR)의 실제 제어 두뇌. gRPC 로 통신. 위치: backend/management/, 포트: 50051.
Ordering Service
고객이 주문을 넣는 웹 페이지(/customer). 외부에 공개.
Ordering Management Service
관리자가 주문을 승인/출고 처리하는 웹 페이지(/admin, /orders). 내부.
canonical (아키텍처)
팀이 합의한 표준 설계도. "canonical 이미지 = 이 프로젝트의 공식 아키텍처 그림".
gRPC
Google 이 만든 고성능 RPC. HTTP/2 위에서 동작. REST 와 다르게 스트리밍(서버가 한 번 응답 시작하면 계속 이벤트를 밀어줄 수 있음)을 지원.
REST vs RPC
REST 는 "자원(Resource) 에 HTTP 메서드" 모델. RPC 는 "함수 호출(procedure call)" 모델. 이 프로젝트는 외부 공개는 REST, 내부는 gRPC.
proxy (프록시)
"대신 전달해주는 역할". Interface 의 POST /api/production/start 가 받은 요청을 Management 로 넘기는 것이 proxy.
싱글톤 (singleton)
프로세스 안에 단 하나만 존재하는 객체. 매번 새로 만들지 않고 처음 만든 것을 재사용.
lazy (지연 생성)
"실제로 처음 필요할 때" 만듦. Management gRPC 채널을 프로세스 시작 시점에 만들지 않고, 첫 호출 때 만드는 것.
SoC (Separation of Concerns)
"관심사 분리". 한 모듈은 한 가지 책임만 가지도록. Interface 는 HTTP, Management 는 설비 제어 — 섞지 않음.
SPOF (Single Point of Failure)
"한 곳이 죽으면 전체가 죽는 지점". Interface 와 Management 를 한 프로세스에 두면 FastAPI 죽으면 공장 정지 — 그게 SPOF.
mTLS
"mutual TLS". 클라이언트와 서버가 서로의 인증서를 검증. 내부 서비스 간 통신을 암호화 + 인증.
smartcast v2 스키마
이 프로젝트의 표준 DB 스키마(27테이블). 테이블 이름이 Ord, Item, EquipTaskTxn 같은 축약형. legacy 는 orders, work_orders 같은 v47 시절 이름.
public schema vs smartcast schema
같은 DB 안에서 테이블을 두 그룹으로 나눔. 새 것은 smartcast.*, legacy 는 public.*. PG 의 search_path 로 기본 조회 대상 조정.
proto / protobuf
gRPC 가 쓰는 "인터페이스 정의 파일". management.proto 를 바탕으로 Python 코드(_pb2.py, _pb2_grpc.py)를 자동 생성.
flag-gated (플래그 잠금)
환경변수로 기능 on/off. 배포하고도 즉시 활성화 안 하고, 안정성 확인 후 flag 만 켜서 전환.

핵심 관찰

  1. 외부 웹(관리자/고객)은 Interface 만 보고, Management 는 존재조차 모른다.
  2. PyQt 는 Management 직결. Interface 를 거치지 않음. 이유: 현장 제어는 Interface 가 죽어도 멈추면 안 됨 (SPOF 회피).
  3. 공장 설비(ROS2·Serial) 와 직접 말하는 건 Management 만. Interface 는 설비 프로토콜 0개.
Interface 는 식당의 홀 매니저, Management 는 주방장. 손님(웹 브라우저)은 홀 매니저에게만 주문 전달. 단, VIP(PyQt 공장 담당자) 는 주방에 직접 들어가 불 끄라고 말할 수 있음. 홀 매니저가 쓰러져도 주방은 계속 돌아가야 공장 가동이 유지됨.

③ 계층 경계 규칙 (C-1 ~ C-5)

이 5개 규칙은 "누가 누구와 말할 수 있나" 를 정의합니다. 깨지면 SoC 가 무너져서 한 곳을 고치면 전혀 엉뚱한 곳이 터지는 재앙이 시작됩니다.

C-1. 웹 포털(고객/관리자)은 Management(:50051) 와 직접 통신하면 안 된다

배경

gRPC 는 기본적으로 HTTP/2 기반이라 브라우저가 직접 못 씁니다(gRPC-Web 같은 변환 레이어 필요). 또한 Management 포트(50051)를 인터넷에 열면 공장 제어 API 가 그대로 노출.

규칙

브라우저 코드(src/) 는 /api/* HTTP 만 호출한다. gRPC 라이브러리 import 금지.

증거

src/lib/api.ts 전체에서 `grpc`·`50051` 참조 0건

왜 중요한가

이 규칙이 깨지면 Management 포트를 방화벽에서 외부로 뚫어줘야 하고, 공격 표면이 급격히 늘어납니다. 또 Management API 스키마 변경 시 브라우저까지 배포해야 해서 릴리즈가 엉킴.

"Admin 대시보드에 실시간 차트를 넣고 싶어서 브라우저에서 gRPC-Web 으로 직접 Mgmt 연결하면 어때요?" → C-1 위반. 대신 Interface 에 WebSocket 엔드포인트 추가 + 서버에서 Mgmt stream 구독 (M-4a 설계 옵션 참조).

C-2. Interface Service 는 HTTP REST 이외의 프로토콜을 외부로 노출하지 않는다

배경

한 프로세스가 HTTP + WebSocket + gRPC 서버 역할을 다 하면, 각 포트별 인증·로깅·모니터링이 파편화됩니다.

규칙

backend/app/main.py 의 FastAPI 앱은 포트 8000 만 바인딩. gRPC servicer 등록 금지.

증거

backend/app/main.py 에서 `grpc.server()` 호출 0건

WebSocket(/ws/dashboard) 은 HTTP Upgrade 방식이라 포트 8000 그대로 사용 — 새 포트 노출 아님. 그래서 허용.

C-3. Interface → Management 채널은 싱글톤 1곳에서만 생성한다

배경

gRPC 채널을 매 요청마다 새로 만들면 TCP handshake + TLS handshake 가 매번 발생해 응답이 수백 ms 씩 늘어납니다. 채널은 "연결 풀" 같은 개념 — 한 번 만들어 재사용.

규칙

코드 어디서든 grpc.insecure_channel(...) 을 직접 호출하면 안 됨. 반드시 app/clients/management.py 의 ManagementClient.get() 싱글톤 경유.

증거

backend/app/ 안에서 `grpc.insecure_channel` 은 clients/management.py 1곳에만 존재

왜 중요한가

채널 누수(close 안 되는 채널이 누적되면) 또는 연결 폭풍(burst traffic 시 채널 N개 생성)은 디버깅이 매우 어렵습니다. 진입점을 1곳으로 강제하면 로그·메트릭·TLS 설정이 한곳에서 관리됨.

C-4. Interface Service 는 ROS2·MQTT·Serial 과 직접 통신하지 않는다

배경

ROS2 는 리눅스 전용(DDS 라이브러리), Serial 은 특정 OS·USB 드라이버 의존. Interface 에 이런 dependency 를 넣으면 AWS 같은 클라우드 배포가 불가능 해짐.

규칙

backend/app/ 하위에 import rclpy, paho, serial 같은 HW 라이브러리 없음. 설비 연결은 Management 가 독점.

증거

backend/app/ 안에 `rclpy`/`paho`/`pyserial` import 0건

Interface 를 "AWS 로 이사할 수 있는 가벼운 짐" 으로 유지. Management 는 공장 바닥에 고정된 "무거운 기계". 둘을 섞으면 Interface 도 공장에 묶임.

C-5. Management 는 Interface HTTP 를 역호출하지 않는다

배경

A → B 는 호출하는데 B → A 도 호출하면 순환 의존. 어느 한쪽을 재시작할 때 데드락 위험, 배포 순서 강제 등 문제 발생.

규칙

Management 안에 requests, httpx, localhost:8000 참조가 없어야 함.

증거

backend/management/ 에 requests/httpx/localhost:8000 참조 0건

canonical 화살표 오해 주의

canonical 이미지의 Interface ↔ Management 양방향 화살표는 "gRPC TCP 연결의 양방향 데이터 흐름" 을 뜻합니다. 같은 TCP 소켓 위에서 Interface 가 호출하고 Management 가 스트리밍으로 응답하는 것 — "Management 가 Interface 에 별도 HTTP 요청을 보내는" 의미가 절대 아닙니다.

④ 내부 구조 규칙 (I-1 ~ I-6)

계층 경계(C 규칙)가 외부 관계라면, I 규칙은 Interface 내부의 견고함에 대한 것입니다.

I-1. Management gRPC 채널은 lazy + 싱글톤

의미

① lazy: FastAPI 가 기동할 때 Management 로 안 붙어도 OK. 첫 호출 때 붙는다. ② 싱글톤: 그 후로는 프로세스가 죽을 때까지 같은 채널 재사용.

왜

Management 가 아직 안 떠있어도 Interface 는 정상 시작해야 함(배포 순서 자유). 그리고 요청마다 채널 만들면 성능 폭망.

I-2. Management 장애 → 503, Interface 는 안 죽는다

의미

Mgmt 가 crash 했거나 응답 안 하면, Interface 는 ManagementUnavailable 예외를 HTTPException(503 Service Unavailable) 로 변환해 브라우저에 반환. Interface 자체는 다른 요청을 계속 처리.

왜

웹 관리자가 "생산 시작" 버튼 눌렀더니 Interface 가 같이 죽어버리면 주문 조회 같은 다른 기능도 마비. 한 기능 실패가 전체 정지로 번지면 안 됨.

이 패턴을 "graceful degradation"(우아한 열화) 이라고 합니다. 하위 시스템 실패가 상위를 끌어내리지 않도록 하는 것.

I-3. proxy 플래그는 모듈 import 시점에 고정

의미

_PROXY_START_PRODUCTION = os.environ.get(...) 가 모듈 최상위에 있고, 함수 안이 아님. 즉 FastAPI 시작할 때 한 번 읽고 그대로 고정.

왜

uvicorn 이 worker 여러 개(보통 2~4개)를 띄우면, 각 worker 가 독립 프로세스. 환경변수를 실시간으로 바꾸면 어떤 worker 는 새 값, 어떤 worker 는 옛 값을 써서 요청마다 결과가 다른 비결정성 버그가 발생. 배포 시 systemctl restart 로 전체 재시작을 강제하면 해결.

"환경변수 export INTERFACE_PROXY_START_PRODUCTION=1 했는데 왜 안 바뀌지?" → worker 재시작 필요. 런북에 명시되어 있음.

I-4. FastAPI lifespan 이 종료 시 채널 정리

의미

@asynccontextmanager async def lifespan(): 의 yield 뒤에 ManagementClient.get().close(). 즉 프로세스 종료 직전에 gRPC 채널을 깨끗이 닫음.

왜

gRPC 채널에 연결된 Mgmt 측 리소스도 정리됨. 안 하면 Mgmt 의 connection count 가 계속 늘어나다가 "too many connections" 로 죽는 누수 발생.

I-5. proto stubs 부재 시 graceful degradation

의미

management_pb2 가 아직 생성 안 된 개발 환경이어도 FastAPI 는 기동. _PROTO_AVAILABLE=False 로 표시하고, Management 호출 시도 시점에만 ManagementUnavailable 예외.

왜

신규 개발자가 make proto 를 잊고 uvicorn app.main:app 해도, 읽기 전용 엔드포인트는 정상 동작. proto 가 필요한 기능만 에러. 최소 작동 가능 상태 (MVP) 를 유지.

I-6. 운영 환경 mTLS 활성화

의미

환경변수 MGMT_GRPC_TLS_ENABLED=1 + certs/ 디렉터리 세팅 시, Interface 와 Management 가 서로 인증서를 교환 검증.

현재 상태

코드 경로는 구현됨. 개발/내부망 환경에서는 0(insecure) 유지. 외부 망에 노출되는 운영 배포 시에만 1.

내부망만 사용하는 공장에서는 현실적으로 mTLS 생략 허용. 외부 노출 시에만 필수.

⑤ 고객 포털 계약 (O-1 ~ O-4)

O-1. 고객 발주는 POST /api/orders/customer 단일 엔드포인트로 수신

의미

고객이 주문 폼 제출하면 이 한 엔드포인트가 받아서 smartcast Ord, OrdDetail, OrdStat(PENDING) 3개 테이블에 동시에 INSERT.

왜 "단일"

단계별(/api/orders/step1, /step2) 로 나누면 중간 단계에서 이탈한 주문이 DB 에 쓰레기로 남음. 한 트랜잭션으로 all-or-nothing.

O-2. 이메일 조회는 빈 배열로 fallback

의미

해당 이메일로 주문이 없어도 404 가 아니라 200 + 빈 배열 [] 반환.

왜

404 는 "경로가 없다" 의 의미. 주문 없음은 "경로는 맞고 결과가 비어 있음". 의미가 다름. 프론트엔드도 빈 배열을 받으면 "주문 없음" 메시지만 띄우면 되지만 404 면 에러 처리 분기가 늘어남.

O-3. 제품 / 하중 등급은 공개 읽기 전용

의미

GET /api/products, GET /api/load-classes 는 인증 없이 조회 가능. 고객이 발주 폼에서 드롭다운 채우는 용도.

O-4. 고객 포털은 발주 상태 변경을 호출하면 안 된다 (현재 🔴 실패)

의미

주문 승인/반려/출고는 관리자만 할 수 있어야 함. 현재 POST /api/orders/{id}/status 에 권한 가드가 없어 아무나 호출 가능.

왜 실패 표시인가

이 엔드포인트가 CORS allow_origins=["*"] + 프론트 password gate 뿐이라, 악의적 사용자가 curl 로 바로 때리면 임의 주문을 "completed" 로 바꿀 수 있음. 운영 배포 전 반드시 해결.

단기 조치: Interface 앞단에 nginx basic auth. 장기: JWT/OAuth 세션 미들웨어 + Depends(require_admin).

⑥ 관리자 포털 계약 (M-1 ~ M-5)

M-1. 주문 파이프라인 6단계 지원

상태 흐름

접수(pending) → 승인(approved) → 생산(in_production) → 생산완료(production_completed) → 출고(shipping_ready) → 완료(completed)

구현

POST /api/orders/{id}/status?new_stat=... 로 전이. smartcast OrdStat 테이블에 이력 INSERT (덮어쓰기 X).

상태 변경은 이력 테이블에 append. 주문의 "현재 상태" 는 "가장 최근 OrdStat row" 로 계산. 이렇게 하면 변경 이력 추적이 공짜로 따라옴.

M-2. 생산 모니터링 실시간 뷰

의미

Admin 포털의 "생산 모니터링" 페이지가 필요한 데이터 — items / equip_tasks / equip_stats — 를 REST 로 제공.

구현

GET /api/production/items, /equip-tasks, /equip-stats. 각각 smartcast ORM 모델과 1:1 매핑.

M-3. 품질 / 물류 / 재고 조회

의미

관리자가 필요한 모든 읽기 는 Interface 에서 바로 DB 조회. Management 를 거치지 않음.

왜 읽기는 Interface 가 직접 하나

① 읽기는 트랜잭션 경합 없음 ② Management 로 proxy 하면 gRPC 왕복 레이턴시 추가 ③ DB 는 공유라 Management 가 쓴 것을 Interface 도 즉시 볼 수 있음.

M-4. 실시간은 소비자 적합형 채널

의미

"모든 UI 에 실시간 push 를 꽂자" 가 아니라, 실시간이 정말 필요한 곳에만 push. 나머지는 polling.

구체적으로

  • PyQt (공장 담당자): 1초 이내 알람 필수 → Management 의 WatchAlerts gRPC 스트림 직접 구독
  • Admin 웹 포털: 비동기 모니터링 → /ws/dashboard 5초 polling 으로 충분
  • Customer 웹: 실시간 필요 없음 → 주문 조회 on-demand
119 대원(PyQt) 은 즉시 호출 무전기. 사무실 직원(Admin) 은 5초마다 새로고침되는 대시보드. 고객(Customer) 은 필요할 때 확인하는 앱. 세 사람 다 실시간 push 를 꽂을 필요 없음.

M-4a (설계 결정). Mgmt → Interface 스트림 relay 는 도입하지 않음

네 가지 근거

  1. 실시간 필요 소비자(PyQt) 는 이미 canonical 직결
  2. Admin 동시접속 ≤ 10 범위에서 5초 polling DB 부하 (6~20 query/s) 가 trivial
  3. stream relay 도입 시 connection mgmt + backpressure + stale client detection 으로 복잡도 상승
  4. canonical 의 bi-dir TCP 화살표는 gRPC full-duplex 프로토콜 속성이며 push 의무 도입 의미 아님

승격 트리거(M-4b)

다음 중 하나 성립 시 Phase C-3 로 승격: ① Critical alert SLA < 2초 요구 ② Admin 동시접속 ≥ 30 상시 ③ 웹에서 현장 제어 UX 필요.

"왜 canonical 에 양방향 화살표가 있는데 구현 안 해?" 라는 질문이 나올 수 있음. 답: 화살표는 "데이터가 양방향으로 흐를 수 있다" 는 프로토콜 사실일 뿐, 어느 방향을 언제 반드시 구현해야 한다 는 명령이 아님.

M-5. 생산 개시는 Management 로 위임

의미

"생산 시작" 은 DB 에 3개 테이블(OrdStat MFG + Item + EquipTaskTxn) 을 원자적으로 INSERT 해야 함. 이 트랜잭션은 Management 가 소유.

구현

Interface 의 POST /api/production/start 는 INTERFACE_PROXY_START_PRODUCTION=1 일 때 Management StartProduction gRPC 로 proxy. 응답 shape 은 기존 클라이언트 호환 유지.

⑦ 데이터 계약 (D-1 ~ D-3)

D-1. Interface 와 Management 는 같은 DB, 같은 URL

의미

두 서비스 모두 DATABASE_URL 환경변수를 공유. 결과적으로 같은 PG 인스턴스의 같은 스키마를 본다.

왜

DB 를 나누면 "어떤 것이 진실인가"(source of truth) 문제가 발생. 같은 주문 데이터가 두 DB 에 있으면 동기화 로직이 필요하고, 불일치 버그가 생김. 공유가 가장 단순.

Management 에만 별도 DB 를 두고 싶은 유혹이 있지만, 그 순간 "Interface 가 쓴 주문이 Management 에 언제 전파되나?" 라는 CDC 문제가 시작됨.

D-2. smartcast v2 스키마가 기본, public schema legacy 는 선별 import

현재 상태

DB 에 두 스키마 병존: smartcast.*(신규 27테이블) + public.*(레거시 잔존 4테이블: alerts, handoff_acks, transport_tasks, rfid_scan_log).

왜 legacy 를 다 못 없앴나

SPEC-AMR-001, SPEC-RFID-001 같은 하드웨어 이벤트 테이블이 smartcast 설계에 아직 안 들어감. 제거 대신 선별 import 로 충돌 회피.

구현

app/models/models_mgmt.py 가 legacy 4 테이블만 명시적으로 re-export. models_legacy.py 를 통째 import 하면 smartcast Item 과 legacy Item 이 SQLAlchemy Base.metadata 에서 충돌.

이사 중인 집 — 새 집(smartcast)으로 옮긴 가구가 대부분이지만, 몇 개(legacy 4 테이블) 는 아직 옛 집에 있음. 필요한 것만 "이 서랍에서 가져와" 라고 명시.

D-3. SQLite 폴백은 없다

의미

DATABASE_URL 환경변수가 없으면 FastAPI 가 fail-fast (시작 시 바로 예외). SQLite 로 자동 전환 안 함.

왜

이전 버전에서는 개발 편의로 SQLite fallback 이 있었는데, 결과적으로 "내 개발 PC 에선 되는데 서버에선 안 됨" 버그가 반복됐다. 같은 DB 엔진(PostgreSQL)만 쓰도록 강제하면 환경 차이 버그가 사라짐.

"로컬 개발에 PG 띄우기 귀찮아요" → Tailscale + 원격 PG (100.107.120.14) 공유. docs/SETUP.md 참조.
