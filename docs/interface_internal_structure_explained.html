Interface Service 내부 구조 — 6개 규칙 해설

§4.2 / I-1 ~ I-6 · 주니어~미드 개발자용 · 2026-04-21

"내부 구조" 란 Interface Service(FastAPI 프로세스) 안에서 어떻게 동작해야 하는가 에 대한 규칙입니다. "외부와 어떻게 통신하나"(C-1~C-5) 와 다르게, 여기서는 내부 리소스(gRPC 채널, 플래그, 예외) 를 어떻게 다룰까 가 핵심.

한 줄 요약: 빨리 뜨고, 오래 안 죽고, 깔끔하게 끝난다.

I-1gRPC 채널은 lazy 생성 + 싱글톤 재사용

"Management 연결선은 처음 필요할 때 딱 하나만 만들고 계속 써라"

무엇을 FastAPI 가 시작할 때 Management 로 바로 연결하지 않고, 첫 gRPC 호출이 발생한 시점에 연결을 생성. 그리고 그 후로는 같은 연결을 프로세스가 죽을 때까지 재사용.
왜
  1. Management 가 아직 안 떠있어도 Interface 는 정상 기동 (배포 순서 자유)
  2. 요청마다 채널 만들면 TCP·TLS 핸드셰이크 때문에 응답이 수백 ms 씩 증가
어디서 clients/management.py:90-104 _ensure_channel() + _instance 싱글톤
class ManagementClient:
    _instance = None

    @classmethod
    def get(cls):                 # 싱글톤 진입점
        if cls._instance is None:
            cls._instance = cls()
            # → __init__ 에서는 채널 안 만듦 (lazy)
        return cls._instance

    def _ensure_channel(self):    # 처음 호출 시에만 채널 생성
        if self._channel is None:
            self._channel = grpc.insecure_channel(...)
전화기를 매번 새로 사서 통화 후 버리는 게 아니라, 한 대 사서 계속 쓰는 것. 처음 전화 걸 때 꺼내서 전원 켜고, 그 후로는 스탠바이.
Interface 기동 순서가 Management 에 의존(Mgmt 먼저 떠야 함) · 요청마다 수백 ms 지연 · 디버깅 불가능한 connection leak

I-2Management 장애 → HTTP 503, Interface 는 안 죽는다

"주방(Mgmt) 이 고장 나도 홀 매니저(Interface) 는 서비스 계속"

무엇을 Management 가 crash, timeout, 연결 거부 등으로 응답 안 하면 ManagementUnavailable 예외 → FastAPI 에서 HTTPException(503 Service Unavailable) 로 변환해 브라우저에 반환. Interface 프로세스는 계속 다른 요청 처리.
왜 "생산 시작" 버튼이 실패해도 "주문 조회" 같은 다른 읽기 기능은 살아있어야 함. 한 기능 장애가 전체 서비스 중단으로 번지면 운영 불가.
어디서 clients/management.py 에서 ManagementUnavailable 발생 → routes/management.py:40 또는 routes/production.py:157-161 에서 503 매핑
# routes/management.py
try:
    client.health()
except ManagementUnavailable as exc:
    raise HTTPException(503, f"Management Service unavailable: {exc}")
비행기 엔진이 하나 꺼져도 다른 엔진으로 회항할 수 있어야 안전. 한 엔진 불량이 기체 전체 추락으로 번지면 안 됨.
이 패턴을 graceful degradation(우아한 열화) 이라고 합니다. 전문용어로 기억해두면 회의에서 편함.

I-3proxy 플래그는 모듈 import 시점 고정

"환경변수는 FastAPI 시작할 때 딱 한 번 읽고 고정"

무엇을 INTERFACE_PROXY_START_PRODUCTION 같은 동작 모드 플래그를 함수 안에서 매번 os.environ.get() 호출하지 않고, 모듈 최상위 상수로 한 번만 읽어 고정.
왜 uvicorn 은 워커(worker) 여러 개를 띄움. 각 워커는 별도 프로세스. 환경변수를 실시간으로 바꾸면 워커마다 다른 값을 써서 요청이 어느 워커로 가느냐에 따라 다른 결과 가 나옴. 그래서 플래그 변경 시 worker 전체 재시작 을 강제해야 일관성 유지.
어디서 routes/production.py:179
# routes/production.py (파일 최상위)
_PROXY_START_PRODUCTION = os.environ.get(
    "INTERFACE_PROXY_START_PRODUCTION", "0"
) in ("1", "true")   # import 시점에 한 번만 평가

@router.post("/start")
def start_production(...):
    if _PROXY_START_PRODUCTION:   # 상수 참조 (재평가 X)
        return _proxy_path(...)
    return _legacy_path(...)
"export FLAG=1 했는데 왜 안 먹혀요?" 질문 반복 발생. 일부 워커만 새 값을 써서 재현하기 어려운 버그 (flaky). 런북에 명시: flag 변경 시 systemctl restart 필수.
영화관 좌석 표: 상영 시작 전에 한 번 끊고 끝. 상영 중간에 좌석 바꾸면 혼란. 서비스 재기동 전에는 그 워커가 받은 값으로 끝까지 간다.

I-4FastAPI lifespan 이 shutdown 시 채널 정리

"끝날 때 쓰던 자원 깔끔하게 닫고 나가기"

무엇을 FastAPI 의 @asynccontextmanager async def lifespan() 안에서 yield 뒤(= 프로세스 종료 직전)에 ManagementClient.get().close() 호출해서 gRPC 채널을 명시적으로 정리.
왜 gRPC 채널을 안 닫고 프로세스 죽이면 Management 측 연결 카운터가 증가한 채 남음. 반복되면 Mgmt 의 max connections 소진 → "too many connections" 로 Mgmt 도 죽음. 리소스 누수.
어디서 main.py:35 (lifespan 함수 내 yield 후)
@asynccontextmanager
async def lifespan(app: FastAPI):
    # ... startup 코드 ...
    yield
    # ↓ shutdown 시 실행
    try:
        ManagementClient.get().close()
    except Exception:
        pass   # 이미 닫혔거나 채널 없을 수도 — 무시
퇴근할 때 사무실 불 끄고 문 잠그기. 안 하면 내일 출근한 사람이 불켜진 빈 사무실 보고 당황.
Python 은 프로세스 종료 시 대부분 자원을 알아서 회수하지만, 원격 리소스(연결 상태) 는 직접 close 하지 않으면 상대편에 "연결됨" 으로 남음.

I-5proto stubs 부재 시 graceful degradation

"자동 생성 파일이 없어도 Interface 는 기동되어야 한다"

무엇을 management_pb2.py, management_pb2_grpc.py (protoc 로 자동 생성되는 파일) 가 없으면 _PROTO_AVAILABLE = False 로 플래그 지정하고 FastAPI 는 그대로 기동. 대신 Mgmt 호출 시도 시 ManagementUnavailable("proto stubs not compiled").
왜 신규 개발자가 처음 git clone 후 uvicorn app.main:app 실행 시, proto 를 아직 안 빌드했어도 읽기 전용 기능은 바로 체험 가능. 모든 기능을 막아버리면 온보딩이 막힘. 최소 작동 가능 상태 (MVP) 를 항상 유지.
어디서 clients/management.py:39-49
try:
    import management_pb2
    import management_pb2_grpc
    _PROTO_AVAILABLE = True
except ImportError as exc:
    logger.warning("proto stubs 미컴파일 — 'make proto' 필요")
    _PROTO_AVAILABLE = False

# 호출 시점에서만 에러
def _ensure_channel(self):
    if not _PROTO_AVAILABLE:
        raise ManagementUnavailable("proto stubs not compiled")
새 차를 샀는데 네비가 아직 활성화 안 됐다고 차가 아예 안 움직이면 곤란. 네비 없이도 주행은 가능. 네비 버튼 누를 때만 "활성화 필요" 안내.

I-6운영 환경 mTLS 활성화

"외부 망에 노출될 때는 서로 신분증 교환"

무엇을 환경변수 MGMT_GRPC_TLS_ENABLED=1 + certs/ 디렉터리에 인증서 세팅 시, Interface↔Management 가 서로의 인증서를 검증. 통신도 TLS 로 암호화.
왜 gRPC 기본은 insecure (평문). 공장 내부망만 쓸 때는 큰 위험 아니지만, VPN 이나 외부 네트워크를 경유하면 중간자 공격(MITM) 가능. mTLS 는 암호화 + 신원 확인 두 가지를 한 번에 해결.
어디서 clients/management.py:97-128 — TLS 코드 경로는 이미 구현되어 있고 환경변수로 토글
if os.environ.get("MGMT_GRPC_TLS_ENABLED", "0") == "1":
    creds = grpc.ssl_channel_credentials(
        root_certificates=ca, private_key=key, certificate_chain=crt
    )
    self._channel = grpc.secure_channel(target, creds)
else:
    self._channel = grpc.insecure_channel(target)  # 내부망 개발
개발/공장 내부망은 현재 MGMT_GRPC_TLS_ENABLED=0 (OFF). 외부 노출 배포 시에만 켜면 됨 (v6_review.md S-001 권장조치 완료).
사무실 내부만 들어가는 엘리베이터는 카드키 없음. 본사 로비(외부 접근 가능 지점) 는 반드시 출입증 검사. 둘 다 완벽히 안전한 건물 만들려면 로비도 카드키 필수지만, 당장 위험 낮은 내부는 미활성 상태로 둘 수 있음.
