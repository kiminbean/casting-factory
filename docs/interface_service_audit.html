① 아키텍처 관계도

본 감사의 대상 구조는 Management Service ← Interface Service ↔ (Ordering Management Service, Ordering Service) 이다. Interface Service 는 3 계층 사이의 유일한 HTTP REST 게이트웨이이며, Management 로는 gRPC(:50051) 단방향 호출 만 수행한다.

* Canonical 이미지는 Interface ↔ Management 사이에 양방향 TCP 화살표를 그린다. 이는 같은 gRPC TCP 연결 위의 데이터 흐름 양방향성을 의미한다 — Interface → Management 는 Health, StartProduction 등 단항 RPC 호출이고, Management → Interface 는 WatchItems, WatchAlerts, WatchCameraFrames 등 server-streaming 응답(이벤트 push)이다. Management 가 별도 HTTP 로 Interface 를 역호출하는 경로는 canonical 에도 없다 (C-5 체크와 일치).

* 구현 현황: Interface→Mgmt 방향은 Phase C-1 (Health) · SPEC-C2 (StartProduction) 로 실체화. Mgmt→Interface stream push 경로는 의도적으로 미도입 (설계 옵션 · §4.4 M-4 · §5 설계 결정 참조) — 현재 웹 Admin/Customer 대시보드는 Interface /ws/dashboard 5초 polling 으로 충분한 반응성 제공. 실시간을 실제로 필요로 하는 PyQt(Monitoring Service) 는 canonical 의 Factory Operator PC ↔ Management TCP 화살표를 따라 WatchItems/WatchAlerts 를 직접 구독한다. Canonical 의 양방향 TCP 는 프로토콜 레벨 full-duplex 를 반영한 것이며, push 의 의무 도입을 의미하지 않는다.

④ 구현 체크리스트 (V6 canonical 기준)

4.1 계층 경계 (Separation of Concerns)

  • C-1. Ordering Service 와 Ordering Management 는 Management Service(:50051) 와 직접 통신하지 않는다. Evidence: src/lib/api.ts 전체에서 grpc/50051 참조 0건. 모든 API 호출이 /api/*(HTTP) 로만 라우팅.
  • C-2. Interface Service 는 HTTP REST 이외의 프로토콜을 외부 소비자에게 노출하지 않는다. Evidence: backend/app/main.py 가 포트 8000 하나만 사용, gRPC servicer 등록 없음.
  • C-3. Interface → Management 화살표는 app/clients/management.py 싱글톤을 통해서만 생성된다. Evidence: 코드베이스 전체에서 management_pb2/grpc.insecure_channel 이 backend/app/ 안에서는 app/clients/management.py 1곳에만 존재.
  • C-4. Interface Service 는 ROS2 · MQTT · ESP32 Serial 과 직접 통신하지 않는다. Evidence: backend/app/ 안에 rclpy/paho/pyserial import 0건.
  • C-5. Management Service 는 Interface 의 HTTP 를 호출하지 않는다 (화살표 단방향). Evidence: backend/management/ 안에 requests/httpx/localhost:8000 참조 0건.

4.2 Interface Service 내부 구조

  • I-1. Management gRPC 채널은 lazy 생성 + 프로세스 lifetime 동안 재사용한다. Evidence: ManagementClient._ensure_channel() (clients/management.py:90-104) + _instance 싱글톤.
  • I-2. Management 장애 시 503 으로 매핑되며 Interface 프로세스는 죽지 않는다. Evidence: ManagementUnavailable → HTTPException(503) (management.py:40, production.py:157-161).
  • I-3. 생산 개시 proxy 플래그가 모듈 import 시점 고정이어야 한다 (SPEC-C2 §11.2). Evidence: _PROXY_START_PRODUCTION = os.environ.get(...) 가 module top-level 상수, production.py:179 에서 참조.
  • I-4. FastAPI lifespan 이 Management 채널을 shutdown 시 정리한다. Evidence: main.py:35 ManagementClient.get().close() in lifespan yield 후.
  • I-5. proto stubs 부재 시 graceful degradation. Evidence: clients/management.py:39-49 _PROTO_AVAILABLE=False + ManagementUnavailable.
  • I-6. mTLS (MGMT_GRPC_TLS_ENABLED=1) 가 운영 환경에서 활성화되어야 한다. Evidence: TLS 코드 경로는 구현 (clients/management.py:97-128) — 현재 default OFF. v6_review.md S-001 에서 ✅ 권장조치 완료로 기록.

4.3 Ordering Service 계약 (고객 포털 요구)

  • O-1. 고객 발주 폼 제출이 POST /api/orders/customer 단일 엔드포인트로 수신된다. Evidence: orders.py:137 @router.post("/customer") + CustomerOrderResponse.
  • O-2. 이메일로 주문 조회 GET /api/orders/lookup?email= 가 빈 배열로 fallback. Evidence: orders.py:244 + src/lib/api.ts:170 통합.
  • O-3. 제품 / EN-124 하중 등급 마스터가 공개 읽기 전용. Evidence: orders.products_router, orders.load_classes_router — main.py:59-60 에서 register.
  • O-4. 발주 상태 변경은 Ordering Service 로부터 수신되지 않는다. Evidence: POST /api/orders/{id}/status 는 admin 포털에서만 호출되도록 권한 가드 필요 — 현재는 CORS allow-all + password gate 만.

4.4 Ordering Management 계약 (관리자 포털 요구)

  • M-1. 주문 파이프라인 6단계 (접수→승인→생산→생산완료→출고→완료) 가 Interface 에서 지원. Evidence: orders.py:274 POST /api/orders/{id}/status?new_stat=... + README §주문 상태 파이프라인.
  • M-2. 생산 모니터링에 필요한 smartcast v2 실시간 뷰 (items, equip_tasks, equip_stats) 가 REST 로 제공. Evidence: production.py:188-212.
  • M-3. 품질 / 물류 / 재고 조회가 REST 로 제공. Evidence: quality.py 11건, logistics.py 6건, dashboard.py.
  • M-4. 실시간 요구가 있는 소비자가 적절한 채널로 이벤트를 수신한다 (소비자 적합형 디자인). Evidence: 실시간을 실제로 필요로 하는 PyQt(Monitoring Service) 는 Mgmt WatchItems/WatchAlerts 스트림을 직접 구독 (canonical Factory Operator PC ↔ Management TCP 화살표). 웹 Admin/Customer 대시보드는 Interface /ws/dashboard 5초 polling 으로 동작 — 비동기 모니터링 특성상 충분한 반응성 제공. Mgmt→Interface stream relay 는 설계 옵션 (§5 참조).
  • M-4a. Mgmt→Interface stream relay 는 현 단계에서 도입하지 않는다 (설계 결정). 근거: ① 실시간 필요 소비자(PyQt) 는 이미 canonical 직접 화살표로 처리됨 ② Admin 동시접속 ≤ 10 범위에서 5초 polling 의 DB 부하 (6~20 query/s) 가 trivial ③ stream relay 도입 시 connection mgmt + backpressure + stale client 처리로 복잡도 상승 ④ canonical 의 양방향 TCP 화살표는 gRPC full-duplex 프로토콜 레벨 속성이며 push 의무 도입을 의미하지 않음.
  • M-4b. 다음 조건 중 하나라도 성립하면 Phase C-3 로 승격하여 push relay 도입. 트리거: ① Critical alert SLA < 2초 요구 명시 · ② Admin 동시접속 ≥ 30 상시 · ③ 웹에서 현장 제어 UX 필요. 도입 시 설계: Interface 가 ManagementClient 로 WatchItems/WatchAlerts 구독 → 인메모리 fan-out → WebSocket relay.
  • M-5. 생산 개시가 Management 로 위임된다 (flag=1 시). Evidence: production.py:170-167 proxy 분기 + DEPLOY §4.

4.5 데이터 계약 / 스키마

  • D-1. Interface 와 Management 는 동일 PG 인스턴스 + 동일 DATABASE_URL 사용. Evidence: backend/.env.local 단일 소스, Mgmt 의 sys.path hack 으로 app.database 재사용.
  • D-2. smartcast v2 스키마 (27 테이블) 가 우선, public schema 의 legacy 4 테이블은 선별 import. Evidence: app/models/models_mgmt.py 에서 Alert · HandoffAck · TransportTask · RfidScanLog 만 re-export (__init__.py).
  • D-3. SQLite 폴백 없음. Evidence: feedback_db_postgresql_only 2026-04-14 이후 정책 · backend 는 DATABASE_URL 미설정 시 fail-fast.
