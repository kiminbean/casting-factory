① 개요

Work Order Scheduler는 공장 관리자가 release한 주문들을 "다음에 무엇을 할까" 순서로 정렬하는 계획 엔진입니다. 실제 실행 결정은 하지 않고, 순위 및 lane 배정만 산출합니다.

1.1 코드베이스

  • backend/app/routes/schedule.py — 15KB, 약 420줄 (SPEC-CTL-001, commit 0952f86)
  • backend/app/models/models.py — ProductionJob, PriorityChangeLog

② 책임 & 경계

2.1 책임

  • work_status = released인 work_order 감시
  • 7요소 가중 점수 계산 (100점 만점)
  • 수동 순서 변경 반영 + 이력 기록
  • lane 속성 확인 (production/shipment)
  • 각 lane의 "다음 active" 선정

2.2 범위 밖

  • 실제 로봇 할당 — Global Task Allocator
  • Task 분해 — Workflow Task Manager
  • 경로/충돌 — Traffic Manager + Resource Reservation Manager

③ 7요소 우선순위 엔진 (100점 만점)

SR-CTL-05 구현. Confluence GUI 페이지와 schedule.py에 명시된 가중치:

요소배점산출 방식
납기일 긴급도 25 D-day 기준: 3일 이내=25, 7일=20, 14일=15, 30일=10, 그 외=0
착수 가능 여부 20 공정 idle + 용해로/조형기/AMR 가용성 (boolean × 20)
주문 체류 시간 15 승인 후 경과일 × 1.5 (최대 15)
지연 위험도 15 예상완료일 - 납기일 마진 분석
고객 중요도 10 주문 금액 상위 20% = 10, 상위 40% = 6, 그 외 = 0
수량 효율 10 수량 적을수록 가점 (빠른 회전)
세팅 변경 비용 5 직전 작업 제품과 동일 제품 연속 시 +5 (보너스)
Tie-break 규칙: 총점 동점 시 FIFO (released_at 오래된 쪽 우선). 완전 동시인 경우 work_order_id 낮은 쪽. 이는 Traffic Manager의 동점 처리와 일관.

④ 감시 루프

Scheduler는 두 가지 방식으로 동작할 수 있습니다:

4.1 Polling (현재 구현)

# 주기적으로 released 상태 work_order 조회
while True:
    time.sleep(POLL_INTERVAL)  # 예: 5초
    released_orders = db.query(WorkOrder).filter_by(
        work_status="released"
    ).all()
    if not released_orders:
        continue
    rankings = calculate_priorities(released_orders)
    apply_rankings(rankings)
    promote_to_queue(rankings)

4.2 Event-driven (향후 개선)

  • PostgreSQL LISTEN/NOTIFY 또는 Redis pub/sub
  • release 버튼 클릭 시 즉시 이벤트 발행 → Scheduler가 즉시 반응
  • 지연 50ms 이하

4.3 Dry-run vs Apply

엔드포인트동작DB 변경
POST /api/production/schedule/calculate점수 계산만, 결과 반환없음 (미리보기)
POST /api/production/schedule/start계산 + lane 큐 삽입released → queued

⑤ Lane 배정 결정

Scheduler는 각 work_order가 어느 lane으로 갈지 결정합니다:

조건LaneCapacity
주조 공정이 필요한 신규 생산 production 1 (순차)
재고에서 바로 출고 (이미 생산된 제품) shipment N (병렬)

5.1 Lane 선택 로직

def decide_lane(work_order) -> str:
    order = work_order.order_item.customer_order

    # 출고 전용 (재고 사용)
    if order.shipment_from_stock:
        return "shipment"

    # 신규 생산이 필요한 경우
    return "production"

from datetime import datetime, timedelta
from fastapi import APIRouter, Depends
from sqlalchemy.orm import Session

router = APIRouter(prefix="/api/production/schedule")

WEIGHT = {
    "due_urgency": 25,
    "start_ready": 20,
    "dwell_time": 15,
    "delay_risk": 15,
    "customer_importance": 10,
    "quantity_efficiency": 10,
    "setup_change": 5,
}

def calculate_priority(order, context) -> dict:
    scores = {}

    # 1) 납기일 긴급도
    days_left = (order.confirmed_due_date - datetime.now().date()).days
    if days_left <= 3: scores["due_urgency"] = 25
    elif days_left <= 7: scores["due_urgency"] = 20
    elif days_left <= 14: scores["due_urgency"] = 15
    elif days_left <= 30: scores["due_urgency"] = 10
    else: scores["due_urgency"] = 0

    # 2) 착수 가능 여부
    scores["start_ready"] = 20 if check_equipment_ready(context) else 0

    # 3) 체류 시간
    dwell_days = (datetime.now() - order.created_at).days
    scores["dwell_time"] = min(dwell_days * 1.5, 15)

    # 4~7) 기타 요소 ...

    total = sum(scores.values())
    return {
        "order_id": order.order_id,
        "total_score": total,
        "breakdown": scores,
        "recommendation_reason": build_reason(scores, order),
    }

@router.post("/calculate")
def calculate_endpoint(req: CalculateRequest, db: Session = Depends(get_db)):
    orders = db.query(CustomerOrder).filter(
        CustomerOrder.order_id.in_(req.order_ids),
        CustomerOrder.order_status == "approved"
    ).all()

    context = build_context(db)  # 설비 상태 등
    rankings = [calculate_priority(o, context) for o in orders]
    rankings.sort(key=lambda r: (-r["total_score"], r["order_id"]))
    return {"ranked": rankings}
