05. Production Lane Queue — 생산 Lane QueueQueue

① 개요

Production Lane Queue는 신규 주조 생산이 필요한 work_order들의 대기열입니다. 한 번에 하나의 주조만 진행 가능하므로 엄격한 순차 큐로 관리됩니다.

1.1 역할

  • Scheduler에서 넘어온 생산 작업을 우선순위 + FIFO 순서로 정렬
  • 한 번에 정확히 1개만 active (capacity=1)
  • 현재 진행 중인 작업이 완료/실패되어야 다음으로 승격

② 왜 capacity=1인가

공장에 용해로 1기, 조형기 1기, 주탕 로봇 1대만 있으면 동시에 2개 주조 불가능. 물리 제약 = capacity 1.

만약 용해로 2기라면? capacity=2로 증가 가능. 이 경우 lane을 여러 개 운영 (production_A, production_B) 또는 capacity를 parametrize.

2.1 물리 자원 분석

③ 자료구조

from dataclasses import dataclass, field
from datetime import datetime
from heapq import heappush, heappop

@dataclass(order=True)
class QueueEntry:
    # 낮은 값이 앞 (heapq는 min-heap)
    sort_key: tuple  # (-priority_score, released_at, work_order_id)
    work_order_id: int = field(compare=False)
    priority_score: int = field(compare=False)
    released_at: datetime = field(compare=False)
    lane: str = field(default="production", compare=False)

@dataclass
class ProductionLaneQueue:
    capacity: int = 1
    queue: list = field(default_factory=list)  # heapq
    active: QueueEntry | None = None

④ Priority Queue 알고리즘

4.1 Sort Key 구성

heapq 삽입 시 튜플 (-priority_score, released_at, work_order_id):

  • -priority_score: 점수 높을수록 앞 (음수 부호로 역전)
  • released_at: 동점 시 먼저 release된 쪽 우선 (FIFO)
  • work_order_id: 완전 동시인 경우 결정론적 tiebreak

4.2 동작 예시

Queue 상태 (높은 우선순위 → 낮은 우선순위): ┌─────────────────────────────────────┐ │ WO-103 score=78 released 10:15 │ ← front (pop 대상) │ WO-105 score=78 released 10:22 │ ← 동점이지만 나중에 released │ WO-101 score=62 released 09:50 │ │ WO-107 score=45 released 11:00 │ └─────────────────────────────────────┘ active: WO-099 (running) WO-099 완료 → pop WO-103 → active Queue: ┌─────────────────────────────────────┐ │ WO-105 score=78 │ │ WO-101 score=62 │ │ WO-107 score=45 │ └─────────────────────────────────────┘

⑤ 상태 전이

work_order.work_status 흐름: [Scheduler 결정] │ ▼ released ──────► queued ──────► running ──────► completed / failed │ ▲ │ │ │ │ │ pop & activate │ │ │ │ └─ (lane queue enqueue) ──────────┘

5.1 전이 규칙

  • released → queued: Scheduler의 apply 단계에서 lane 큐 삽입
  • queued → running: active가 비면 pop + 승격
  • running → completed: 모든 task 성공
  • running → failed: 복구 불가, Execution Monitor 판단

⑥ 구현 스케치

class ProductionLaneQueue:
    CAPACITY = 1

    def __init__(self, db_session):
        self.db = db_session
        self._heap: list[QueueEntry] = []
        self._active: QueueEntry | None = None
        self._lock = threading.RLock()

    def enqueue(self, work_order) -> None:
        """Scheduler가 우선순위 계산 후 호출."""
        with self._lock:
            entry = QueueEntry(
                sort_key=(
                    -work_order.priority_score,
                    work_order.released_at,
                    work_order.work_order_id,
                ),
                work_order_id=work_order.work_order_id,
                priority_score=work_order.priority_score,
                released_at=work_order.released_at,
            )
            heappush(self._heap, entry)

            # DB 상태: released → queued
            work_order.work_status = "queued"
            work_order.queue_order = len(self._heap)
            self.db.commit()

            # 즉시 activate 시도
            self._try_activate()

    def _try_activate(self) -> None:
        """active가 비어있으면 queue front를 올림."""
        if self._active is not None:
            return  # 이미 running 중
        if not self._heap:
            return  # 큐 비었음

        entry = heappop(self._heap)
        self._active = entry

        wo = self.db.query(WorkOrder).get(entry.work_order_id)
        wo.work_status = "running"
        wo.actual_start_at = datetime.utcnow()
        self.db.commit()

        # Workflow/Task Manager에게 task 분해 요청
        emit_event("work_order.activated", work_order_id=wo.work_order_id)

    def complete_active(self, success: bool) -> None:
        """Execution Monitor가 호출."""
        with self._lock:
            if self._active is None:
                return
            wo = self.db.query(WorkOrder).get(self._active.work_order_id)
            wo.work_status = "completed" if success else "failed"
            wo.actual_end_at = datetime.utcnow()
            self.db.commit()
            self._active = None
            self._try_activate()  # 다음 work_order 승격

    def peek(self) -> list[dict]:
        """UI 조회용. 큐 스냅샷."""
        with self._lock:
            return [
                {
                    "work_order_id": e.work_order_id,
                    "priority_score": e.priority_score,
                    "position": i + 1,
                }
                for i, e in enumerate(sorted(self._heap))
            ]

⑦ 동시성 처리

7.1 경쟁 조건

  • 동시 enqueue: 여러 release가 거의 동시 → lock으로 직렬화
  • active 중 enqueue: 새 고우선순위 주문이 왔을 때 active를 중단할 것인가? → 아니오. 진행 중 작업은 완료까지 보호 (preemption 금지)
  • 서버 재시작: DB에서 queued/running 레코드를 읽어 heap 재구성

7.2 영속성

in-memory heap은 보조 인덱스일 뿐, 진실 원천은 DB. 서버 재시작 시:

def recover_from_db(self):
    self._heap.clear()
    self._active = None

    # running 복구
    running = self.db.query(WorkOrder).filter_by(
        lane="production",
        work_status="running",
    ).first()
    if running:
        self._active = QueueEntry(...)

    # queued 복구
    queued = self.db.query(WorkOrder).filter_by(
        lane="production",
        work_status="queued",
    ).order_by(WorkOrder.queue_order).all()

    for wo in queued:
        heappush(self._heap, QueueEntry(...))
