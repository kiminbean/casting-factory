① 개요

Office UI는 주물 공장의 고객과 사무실 직원이 사용하는 웹 기반 발주·검토·승인 인터페이스입니다. 아키텍처에서 가장 상위의 프레젠테이션 계층이며, 모든 주문은 여기서 시작됩니다.

  • src/app/customer/orders/page.tsx — 고객 주문 조회
  • src/app/orders/page.tsx — 사무실 직원 주문 관리
  • src/components/Sidebar.tsx — 관리자 네비게이션

② 역할 & 범위

2.1 책임 (Responsibilities)

  • 주문 입력 수집: 고객이 제품·옵션·수량·납기·고객정보를 입력할 수 있는 폼 제공
  • 입력 검증: 클라이언트 측 즉시 검증 (최소/최대 값, 필수 항목)
  • 견적 계산: 실시간 예상 합계 금액 표시
  • 주문 상태 조회: 고객이 과거 주문 상태를 열람
  • 사무실 검토: 직원이 접수된 주문을 검토하고 승인/반려 결정

2.2 범위 밖 (Out of Scope)

  • 생산 개시 버튼 — Factory Manager UI가 담당
  • 우선순위 계산 — Work Order Scheduler 담당
  • DB 직접 접근 — 반드시 Order API 경유 (Interface Layer 규칙)
  • 로봇/설비 제어 — 별도 계층

④ 5단계 발주 폼 (/customer)

SR-ORD-01-06 "주문 절차는 최대 5단계 이내" 요구사항 준수.

Step화면입력 필드검증
1 제품 선택 카테고리 필터 + 제품 카드 (이미지/이름/재질/기본 가격) 제품 1개 선택 필수
2 규격 입력 직경, 두께, 하중 등급, 재질 최소/최대 범위 실시간 검증 (SR-ORD-01-04)
3 후처리 / 견적 아이콘 카드 (버 제거, 그라인딩) + 디자인 시안 + 자동 견적 예상 금액 실시간 계산
4 고객 정보 회사명, 담당자, 연락처, 이메일, 배송지, 희망 납기, 비고 모든 필드 필수 (SR-ORD-01-06)
5 주문 완료 전체 요약 미리보기 → 최종 제출 주문 ID 부여 후 상태 received로 저장

⑤ 주문 상태 전이 (6단계)

SR-ORD-02 "6단계 상태" 요구사항. Office UI는 received → reviewing → approved/rejected 구간을 담당합니다.

[고객] [사무실 직원] [공장 관리자] │ │ │ ▼ ▼ ▼ received ─────────► reviewing ───► approved ────► in_production │ │ │ │ │ │ │ ▼ │ │ │ ready_to_ship │ │ │ │ │ │ │ ▼ │ │ │ completed │ │ │ │ └──► rejected │ └─── (고객은 received~reviewing 상태 조회만 가능)
상태 전이 규칙:
  • received: 고객 제출 시 자동
  • reviewing: 직원이 검토 시작하면 전환
  • approved: 직원 승인 → Office UI 역할 종료, 이후 Factory Manager UI로 넘어감
  • rejected: 직원 반려 → 고객에게 알림

⑥ API 연동

Office UI가 호출하는 REST API (Order API 참조):

MethodEndpoint용도
GET/api/products제품 마스터 목록 (Step 1)
POST/api/orders신규 주문 생성 (Step 5 제출)
GET/api/orders전체 주문 목록 (상태 탭 필터링)
GET/api/orders/{id}/details주문 상세 조회
PATCH/api/orders/{id}/status승인/반려 (직원)
PATCH/api/orders/{id}견적/납기 수정 (직원)
GET/api/alerts알림 배너용

7.1 5단계 폼 컴포넌트 (단순화)

// src/app/customer/page.tsx
"use client";
import { useState } from "react";

type OrderDraft = {
  product_id: number | null;
  diameter: number;
  thickness: number;
  load_grade: string;
  material: string;
  post_processing: string[];
  quantity: number;
  company_name: string;
  manager_name: string;
  phone: string;
  email: string;
  shipping_address: string;
  requested_due_date: string;
  memo: string;
};

export default function CustomerOrderPage() {
  const [step, setStep] = useState<1|2|3|4|5>(1);
  const [draft, setDraft] = useState<OrderDraft>(initialDraft);
  const [estimatedPrice, setEstimatedPrice] = useState(0);

  const handleSubmit = async () => {
    const res = await fetch("/api/orders", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify(draft),
    });
    if (res.ok) {
      const { order_id } = await res.json();
      router.push(`/customer/orders?created=${order_id}`);
    }
  };

  return (
    <div>
      <StepIndicator current={step} total={5} />
      {step === 1 && <ProductSelect onNext={() => setStep(2)} ... />}
      {step === 2 && <SpecInput onNext={() => setStep(3)} ... />}
      {step === 3 && <PostProcess onNext={() => setStep(4)} ... />}
      {step === 4 && <CustomerInfo onNext={() => setStep(5)} ... />}
      {step === 5 && <ReviewSubmit onSubmit={handleSubmit} ... />}
    </div>
  );
}

// src/app/orders/page.tsx (간략)
async function approveOrder(orderId: string, confirmedPrice: number) {
  await fetch(`/api/orders/${orderId}/status`, {
    method: "PATCH",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      order_status: "approved",
      total_confirmed_price: confirmedPrice,
    }),
  });
  // → Order API가 상태 전이 검증 후 DB 저장
  // → Factory Manager UI에서 조회 가능해짐
}

⑨ 통합 포인트

상류 (Upstream)

  • 고객: 웹 브라우저를 통해 직접 사용
  • 사무실 직원: 사이드바 네비게이션에서 /orders 접근

하류 (Downstream)

  • Order API / DB: 모든 데이터 영속화
  • Factory Manager UI: 승인 후 release 단계로 넘김 (간접 연결, DB 경유)

상태 소유권

Office UI는 주문 상태를 소유하지 않음. 항상 API 호출로 상태를 변경하고, 로컬은 표시용 캐시만 보유. received → reviewing → approved/rejected 전이는 DB 레벨에서 검증됨.
