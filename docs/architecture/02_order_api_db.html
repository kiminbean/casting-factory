① 개요

Order API / DB는 주문의 전 생명주기를 단일 진실 원천(Single Source of Truth)으로 관리합니다. 모든 UI는 이 계층을 통해서만 주문 데이터를 변경할 수 있으며, 상태 머신의 규칙을 API가 강제합니다.

1.1 기술 스택

  • Framework: FastAPI (Python 3.11)
  • Database: PostgreSQL 16 + TimescaleDB (시계열 로그용)
  • ORM: SQLAlchemy 2.0
  • Migration: Alembic
  • WebSocket: 5초 주기 대시보드 브로드캐스트

1.2 코드베이스 위치

  • backend/app/main.py — FastAPI 엔트리
  • backend/app/database.py — DB 연결, 세션
  • backend/app/models/models.py — SQLAlchemy 모델
  • backend/app/routes/ — 8개 라우터 모듈
  • backend/app/schemas/ — Pydantic 스키마
  • backend/app/seed.py — 시드 데이터

② 상태 머신 (work_order)

아키텍처의 핵심은 work_order의 명확한 상태 전이입니다. 사용자 요청 다이어그램에서 명시된 4가지 내부 상태(approved/released/queued/running)를 work_status 컬럼에 반영합니다.

2.1 전이도

[고객] [사무실 직원] [공장 관리자] │ │ │ ▼ ▼ ▼ customer_order: received ────► reviewing ─► approved (release 버튼) │ │ └─ rejected │ │ work_order: waiting ─── released ───►queued ──► running ──► completed │ │ └── failed ─────┘

2.2 work_order 상태 정의

상태의미진입 조건다음 상태
waiting주문 승인 후 생산 대기customer_order.status = approvedreleased
released공장 관리자가 생산 개시 허가Factory Manager UI 버튼 클릭queued
queuedScheduler가 lane 큐에 삽입Work Order Scheduler가 승격running / released(취소)
running공정 실행 중lane active 선정completed / failed
completed모든 공정 완료마지막 task 성공(종료)
failed복구 불가 실패Execution Monitor 판단(종료 또는 수동 재시도)
핵심: approved → released는 자동 승격 아님. 공장 관리자가 명시적으로 버튼을 눌러야 release 전이. 이는 현장 준비 상태(자재/패턴/로봇 가용성)를 사람이 판단하기 위함.

3.2 work_order 핵심 컬럼

CREATE TABLE work_order (
    work_order_id     SERIAL PRIMARY KEY,
    order_item_id     INT REFERENCES order_item(order_item_id),
    priority_score    INT,                    -- Scheduler가 채움
    priority_rank     INT,                    -- lane 내 순위
    queue_order       INT,                    -- 큐 삽입 순서
    lane              VARCHAR(20),             -- 'production' | 'shipment'
    work_status       VARCHAR(20) NOT NULL,
    due_date          DATE,
    released_at       TIMESTAMP,              -- 관리자 버튼 클릭 시점
    planned_start_at  TIMESTAMP,
    actual_start_at   TIMESTAMP,
    actual_end_at     TIMESTAMP,
    assigned_to       VARCHAR(50)              -- 로봇/설비 ID
);

CREATE INDEX idx_work_order_lane_status ON work_order(lane, work_status);
CREATE INDEX idx_work_order_due_date ON work_order(due_date);

⑤ 상태 전이 로직

API 레벨에서 전이 규칙을 강제하여 잘못된 상태 이동을 차단합니다.

# backend/app/services/work_order_service.py (sketch)
from enum import Enum

class WorkStatus(str, Enum):
    WAITING = "waiting"
    RELEASED = "released"
    QUEUED = "queued"
    RUNNING = "running"
    COMPLETED = "completed"
    FAILED = "failed"

# 허용된 전이 그래프
ALLOWED_TRANSITIONS = {
    WorkStatus.WAITING:   {WorkStatus.RELEASED},
    WorkStatus.RELEASED:  {WorkStatus.QUEUED, WorkStatus.WAITING},  # 취소 허용
    WorkStatus.QUEUED:    {WorkStatus.RUNNING, WorkStatus.RELEASED}, # 재배치
    WorkStatus.RUNNING:   {WorkStatus.COMPLETED, WorkStatus.FAILED},
    WorkStatus.COMPLETED: set(),
    WorkStatus.FAILED:    {WorkStatus.QUEUED},  # 수동 재시도
}

def transition(work_order, new_status: WorkStatus, actor: str, reason: str = ""):
    current = WorkStatus(work_order.work_status)
    if new_status not in ALLOWED_TRANSITIONS[current]:
        raise HTTPException(
            409,
            f"Invalid transition {current} → {new_status}",
        )

    old = work_order.work_status
    work_order.work_status = new_status.value

    # 타임스탬프 자동 기록
    now = datetime.utcnow()
    if new_status == WorkStatus.RELEASED:
        work_order.released_at = now
    elif new_status == WorkStatus.RUNNING:
        work_order.actual_start_at = now
    elif new_status in (WorkStatus.COMPLETED, WorkStatus.FAILED):
        work_order.actual_end_at = now

    # 이력 기록
    db.add(PriorityChangeLog(
        work_order_id=work_order.work_order_id,
        old_status=old,
        new_status=new_status.value,
        reason=reason,
        changed_by=actor,
        changed_at=now,
    ))
    db.commit()

    # WebSocket 브로드캐스트 트리거
    broadcast_work_order_update(work_order)

⑥ Release 개념

사용자 다이어그램에서 명시된 "release" 단계는 이 프로젝트의 사람-기계 경계선입니다.

Release가 필요한 이유: 사무실에서 주문을 승인해도(approved), 공장 현장이 준비되지 않았을 수 있음. 자재 수급, 패턴 준비, 로봇 가용성 등을 사람이 최종 판단하고 "이제 실제로 생산해도 된다"는 신호를 보내는 것이 release.

6.1 Release 시점 검증 (API 측)

사용자가 release 버튼을 누르면 API가 다음을 검증합니다:

  • customer_order.status == approved
  • work_order 생성 여부 확인 (자동 생성 or 수동)
  • 예약 중복 방지 (이미 released면 409)
  • 타임스탬프 released_at 기록

⑦ 구현 스케치 (FastAPI)

# backend/app/routes/schedule.py (excerpt)
from fastapi import APIRouter, Depends, HTTPException
from sqlalchemy.orm import Session

router = APIRouter(prefix="/api/production/schedule")

@router.post("/start")
def start_production(
    req: StartProductionRequest,
    db: Session = Depends(get_db),
    user: User = Depends(require_operator),
):
    """
    공장 관리자 release 버튼 처리.
    approved 상태의 customer_order들을 work_order로 변환하고
    work_status를 released로 전이.
    """
    results = []
    for order_id in req.order_ids:
        order = db.query(CustomerOrder).filter_by(order_id=order_id).first()
        if not order:
            raise HTTPException(404, f"Order {order_id} not found")
        if order.order_status != "approved":
            raise HTTPException(
                409,
                f"Order {order_id} not approved",
            )

        # work_order 생성 또는 조회
        wo = db.query(WorkOrder).filter_by(
            order_item_id=order.order_item_id
        ).first()
        if not wo:
            wo = WorkOrder(
                order_item_id=order.order_item_id,
                work_status="waiting",
                lane=req.lane,
            )
            db.add(wo)
            db.flush()

        # 상태 전이
        transition(wo, WorkStatus.RELEASED, actor=user.user_name,
                   reason="manager_release")

        results.append({
            "work_order_id": wo.work_order_id,
            "status": wo.work_status,
        })

    db.commit()
    return {"released_count": len(results), "work_orders": results}

⑧ NFR & 트랜잭션

항목기준
API 응답 시간 (단일 조회)p95 ≤ 200ms
상태 전이 원자성DB 트랜잭션 보장 (READ COMMITTED 이상)
동시 release 경쟁SELECT FOR UPDATE로 work_order 락
WebSocket 브로드캐스트5초 주기 (폴링 기반), 실시간 필요 시 즉시 push
DB 백업일 1회 pg_dump + TimescaleDB 압축
마이그레이션Alembic, down-revision 포함
주의: 이전 설계에서 SQLite를 명시한 문서가 일부 있으나, 실제 코드베이스는 PostgreSQL 16 + TimescaleDB로 마이그레이션 완료/진행 중. Confluence GUI 페이지 하단의 "SQLite" 기재는 오래된 정보 (docs/CONFLUENCE_FACTS.md 불일치 표 참조).
