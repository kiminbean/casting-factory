8 V6 결정 (2026-04-14)

Management Service 독립 프로세스화 + gRPC 직결

합의 사항: Factory Operator PC(PyQt)는 Interface Service(FastAPI)를 거치지 않고 Management Service(gRPC :50051) 와 직접 TCP 통신한다.
항목결정
프로토콜gRPC (Protocol Buffers)
포트50051
네트워크공장 내부망 전용 (외부 방화벽 차단)
PyQt 연결Management Service 직결
Admin/Customer PCInterface Service(HTTP) 유지
DBInterface Service 와 공유 (PostgreSQL + TimescaleDB)
이유: Interface Service 장애/AWS 이관 중에도 공장 가동을 중단시키지 않기 위함. 현장 PC 는 Interface Service 의 배포·장애·네트워크 변경에 영향받지 않는다.

상세 설계 문서: docs/management_service_design.md
gRPC API 정의: backend/management/proto/management.proto
서버 스캐폴드: backend/management/
PyQt 클라이언트: monitoring/app/management_client.py

1 검증 방법론

이전 분석의 문제점

v1 보고서에서 Fleet State Manager를 P0 필수로 제안했으나, 프로젝트 코드베이스의 equipment 테이블이 이미 로봇 위치/배터리/상태를 관리하고 있었음. 일반론적 FMS 아키텍처를 프로젝트 실정 확인 없이 적용한 오류.

v2 검증 원칙: 모든 컴포넌트 제안은 반드시 (1) Confluence SR ID와 (2) 코드베이스 실제 구현 상태를 교차 확인한 후 판정.

7 구현 우선순위 및 로드맵

우선순위컴포넌트작업 유형추정 규모근거
P0 Robot / Cell Executor 신규 구축 ROS2 패키지 + Python 인터페이스 이것 없이는 어떤 Task도 실행 불가. SA V6의 5개 Controller 구현체
P1 Execution Monitor 신규 구축 백엔드 서비스 (비동기 감시 루프) SR-CTL-03 이상 감지 1초 이내. 무인 운영의 필수 안전망
P1 Traffic Manager 설계 → 구현 Waypoint 그래프 + 충돌 검사 엔진 AMR 2대 이상 동시 이동 시 필수. 설계는 완료
P2 Task Manager 보강 기존 코드 확장 schedule.py에 Task 분해 로직 추가 현재 수동 분해 → 자동화
P2 Task Allocator 보강 기존 코드 확장 스코어링 함수 개선 거리 + capability + 배터리 종합 평가
P3 로봇 상태 보강 API 1개 추가 heartbeat 엔드포인트 + offline 로직 equipment 테이블 기존 활용
P3 적재 관리 보강 로직 추가 스테이션 예약 + FIFO/LIFO 선택 warehouse_racks 기존 활용
