Fleet Traffic Management — 한눈에 보는 요약

우리 공장 AMR 3대가 서로 부딪히지 않고 움직이는 방법 · 2026-04-08
한 문장 요약: 공장 맵을 점(waypoint)과 선(edge)의 지도로 만들고, 로봇은 이동할 선을 미리 예약해서 움직이며, 두 로봇이 서로 막히면 우선순위가 낮은 로봇이 뒤로 한 칸 비켜주는 방식으로 충돌과 교착을 방지합니다.
맵 크기
1m × 2m
로봇 크기
0.12 × 0.12m
로봇 대수
AMR 3대
구동 방식
차동 구동

① 해결하려는 문제

공장에 로봇이 여러 대 있으면 자연스럽게 이런 상황이 생깁니다:

  • 경로 충돌 — 두 로봇이 같은 복도를 쓰려고 함
  • 교착(Deadlock) — 두 로봇이 좁은 복도에서 정면으로 마주쳐 둘 다 멈춤
  • 비효율 — 한 로봇이 작업 중인데 다른 로봇이 끝없이 기다림

Nav2(로봇 내비게이션 라이브러리)만으로는 로봇 한 대의 이동은 잘 하지만, 여러 로봇이 서로를 알고 조율하는 일은 못 합니다. 그걸 우리가 추가로 만들어야 합니다.

② 핵심 개념 3가지

🗺️

1. Waypoint + Edge 지도

공장을 "주요 지점(waypoint)"과 "그 사이를 잇는 길(edge)"로 간단한 그래프로 표현합니다. 예: 용해로 — 교차점 — 검사대

🔒

2. Edge 예약

로봇이 길을 지나가려면 그 edge를 먼저 "내 것"이라고 예약합니다. 다른 로봇이 이미 예약했으면 다른 길로 돌아갑니다.

🤝

3. 우선순위 양보

정말 다른 길이 없어서 두 로봇이 막혔을 때, 우선순위가 낮은 로봇이 뒤로 한 칸 비켜줍니다. 점수가 같으면 나중에 온 쪽이 양보합니다.

③ 우리 공장 지도 예시

WP_MELT ─── WP_INT_L ═════════ WP_INT_R ─── WP_INSP (용해로) (좌측 교차점) │ │ (우측 교차점) (검사대) │ │ └───┴── WP_CHARGE (충전소) ● 점(waypoint): 로봇이 멈추거나 작업하는 지점 ─ 선(edge): 로봇이 이동하는 길. 한 번에 1대만 통과
초보자 팁: 복잡한 수학이나 공간 분할이 필요 없습니다. 그냥 "중요한 위치 5~10개를 찍고 선으로 연결한 지도"입니다. 지하철 노선도와 똑같은 구조예요.

④ 실제 동작 시나리오 — AMR-A와 AMR-B

상황: AMR-A가 용해로 → 검사대로 이동 중. 반대편에서 AMR-B도 검사대 → 용해로로 이동해야 함.

  1. AMR-A 출발: 용해로 → 좌측교차점 → 긴 복도 → 우측교차점 → 검사대. 지나갈 edge들을 하나씩 예약하며 진행.
  2. AMR-B가 출발하려 함: 최단 경로는 AMR-A와 정반대 방향. 긴 복도에서 마주칠 예정.
  3. 자동 우회: 시스템이 AMR-A의 예약을 보고, AMR-B에게는 충전소를 거쳐 가는 우회 경로를 자동으로 배정. 마주칠 일이 없음.
  4. 만약 우회로도 없다면: 둘이 복도에서 만나고 3초 기다려도 못 지나가면 교착으로 판정.
  5. 우선순위 비교: AMR-A가 긴급 작업이라 점수가 높음 → AMR-A가 winner, AMR-B가 loser.
  6. AMR-B 양보: 지나온 마지막 waypoint(한 칸 뒤)로 후진하여 비켜줌.
  7. AMR-A 통과 후: AMR-B가 다시 원래 목적지로 출발.

⑦ 앞으로의 진행 순서

  1. 설치 확인: ROS 2 Jazzy에 nav2_route 패키지가 설치 가능한지 확인
  2. 지도 작성: factory_graph.geojson 파일에 우리 공장 waypoint와 edge 정의
  3. TrafficManager 구현: edge 예약·해제 로직 (약 150줄)
  4. Bridge 노드: 예약 상태를 Nav2에 전달하는 ROS2 노드 (약 50줄)
  5. Priority Yield 구현: 교착 감지와 양보 로직 (약 200줄)
  6. 통합 테스트: Gazebo 시뮬레이션 → 실제 AMR 3대 테스트
주의 사항:
  • nav2_route는 Jazzy 이후에만 있음 — 이전 버전에서는 source build 필요
  • 설치 불가 시 대안: Dijkstra를 직접 구현 (약 80줄 추가)
  • 우리 문제 규모(3대 AMR, 1m×2m)라서 복잡한 fleet management(RMF 등)는 불필요
