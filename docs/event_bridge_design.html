핵심 질문 답: Event Bridge 가 DB 에서 정보를 읽고 다른 컴포넌트에 전달합니까?

결론: 아니요. Event Bridge 는 DB 를 직접 읽지 않습니다.
  • SSOT 원칙: DB 읽기·쓰기는 State Manager 가 단독 책임. Confluence 41648180 "모든 판단은 DB(State Manager)에 저장된 상태를 기준으로"
  • Event Bridge 의 실제 일: "발행된 이벤트를 받아서, 다음에 누구를 호출할지 결정 → 호출"
  • 정보 전달 방식: DB 읽기 아님. 이벤트 페이로드 또는 State Manager 래퍼 호출
예외적으로 허용: 라우팅 판단용 읽기 전용 메타 조회
  • 예: TASK_COMPLETED 이벤트를 받았을 때 "다음 공정 type 이 뭔지" 조회 필요 → State Manager 의 읽기 전용 래퍼 호출 OK
  • 조건: ① read-only, ② 조회 결과를 기반으로 create_task() 또는 publish_event() 만 호출
  • DB write 는 절대 불가 (Executor / TaskManager / StateManager 만 write 가능)

왜 이 구분이 중요한가

패턴문제
Event Bridge 가 DB 에 직접 접근 (anti-pattern) StateManager 이외 경로로 상태 변경 가능 → SSOT 깨짐, race condition, 감사 추적 어려움
Event Bridge 가 여러 컴포넌트 세부 로직까지 담당 (anti-pattern) God Object. 의존성 폭증. 변경 시 모든 테스트 깨짐
Event Bridge 가 순수 라우터 (권장) 입력=이벤트, 출력=컴포넌트 호출. 단위 테스트 용이. 추가/제거가 안전

2. Event Bridge 책임 범위

2.1 하는 것 (DO)

  • ✅ 이벤트 수신 (on_event(event))
  • ✅ 이벤트 타입 → 수신 컴포넌트 매핑
  • ✅ 수신 컴포넌트의 on_task_completed 같은 핸들러 호출
  • ✅ 라우팅 결정을 위한 읽기 전용 State Manager 조회
  • ✅ 이벤트 로깅 / 감사 추적

2.2 하지 않는 것 (DON'T)

  • ❌ DB 직접 read / write (SQL, ORM 접근)
  • ❌ 실제 장비 명령 발행 (→ Robot Executor 몫)
  • ❌ 설비 배정 결정 (→ Task Allocator 몫)
  • ❌ 장비 프로토콜 변환 (→ Adapter 몫)
  • ❌ 이벤트 스스로 발생시키기 (Executor 만 발행 — 41648180 고정 규칙)

7.2 Protocol (42205202 네이밍 규약 준수)

from __future__ import annotations
from typing import Protocol, runtime_checkable, Callable
from enum import Enum
from pydantic import BaseModel
from datetime import datetime


class EventType(str, Enum):
    TASK_CREATED   = "TASK_CREATED"
    TASK_STARTED   = "TASK_STARTED"
    TASK_COMPLETED = "TASK_COMPLETED"
    TASK_FAILED    = "TASK_FAILED"
    HANDOFF_ACK    = "HANDOFF_ACK"
    RFID_SCANNED   = "RFID_SCANNED"
    STATE_CHANGED  = "STATE_CHANGED"


class Event(BaseModel):
    event_type:  EventType
    txn_id:      int | None = None
    item_id:     int | None = None
    resource_id: str | None = None
    payload:     dict        = {}
    occurred_at: datetime


@runtime_checkable
class IEventBridge(Protocol):
    def publish(self, event: Event) -> None: ...
    def subscribe(self, event_type: EventType, handler: Callable[[Event], None]) -> None: ...


class EventBridgeImpl:
    """
    In-process event dispatcher.

    Side Effects 목록:
      publish()  → 이벤트 로깅 (감사) + subscribe 된 모든 handler 호출
      subscribe() → self._handlers[event_type].append(handler)

    허용 DB 접근:
      - StateManager.read_* (read-only)
    금지 DB 접근:
      - 직접 cursor / ORM write
    """
    def __init__(self, state_manager, logger) -> None:
        self._state = state_manager      # 오직 read-only 래퍼 용도
        self._log = logger
        self._handlers: dict[EventType, list[Callable]] = {}

    def publish(self, event: Event) -> None:
        self._log.info("Event published: %s", event.event_type)
        for handler in self._handlers.get(event.event_type, []):
            try:
                handler(event)
            except Exception:
                self._log.exception("handler failed for %s", event.event_type)

    def subscribe(self, event_type: EventType, handler) -> None:
        self._handlers.setdefault(event_type, []).append(handler)

7.3 결선 예시 (startup 시 구독 등록)

def wire_bridge(bridge, task_mgr, allocator, state_mgr, amr_fsm):
    # TASK_CREATED → Allocator
    bridge.subscribe(EventType.TASK_CREATED,
                     lambda e: allocator.assign_task(e.txn_id))

    # TASK_COMPLETED → 다음 공정 생성
    def on_completed(e: Event):
        task = state_mgr.get_task(e.txn_id)
        next_type = state_mgr.get_next_task_type(task)
        if next_type:
            task_mgr.create_task(TaskCreateInput(
                ord_id=task.ord_id, item_id=task.item_id, task_type=next_type,
            ))
    bridge.subscribe(EventType.TASK_COMPLETED, on_completed)

    # TASK_FAILED → 설비 ERROR 마킹
    bridge.subscribe(EventType.TASK_FAILED,
                     lambda e: state_mgr.mark_error(e.resource_id, e.payload.get("error_code")))

    # HANDOFF_ACK → AMR FSM
    bridge.subscribe(EventType.HANDOFF_ACK,
                     lambda e: amr_fsm.confirm_handoff(e.resource_id))

7.4 테스트 관점

  • 단위 테스트: publish(Event) → 등록된 handler 가 정확히 호출되는지만 검증 (mock handler)
  • State Manager, Task Manager 는 모두 mock. Event Bridge 는 순수 라우터라 DB 의존 없이 테스트 가능
  • 회귀 방지: 새 이벤트 타입 추가 시 EventType enum 에만 추가하면 되고 handler 등록 누락은 lint 규칙으로 검출 가능

8. 현재 코드베이스 대응 관계

Event Bridge 가 아직 없으므로 이벤트 라우팅 기능은 여러 파일에 분산됨. 도입 시 흡수 대상:

현재 위치하는 일Event Bridge 로 이동 시
server.py:240 ReportHandoffAck handoff_acks INSERT + transport_tasks UPDATE + FSM 전이 + WebSocket notify DB write 는 State Manager 로, FSM 전이 + WebSocket 은 Event Bridge 의 HANDOFF_ACK 핸들러로
services/execution_monitor.py SLA 타임아웃 polling + Alert INSERT polling 은 유지, Alert 생성은 State Manager 경유 + TASK_FAILED 이벤트 발행
services/fms_sequencer.py 공정 순서 orchestration TaskEvent 구독으로 대체 — sequencer 는 Event Bridge 에 handler 등록
services/amr_state_machine.py FSM 전이 FSM 은 State Manager 소속으로 편입. Event Bridge 는 "FSM 전이 이벤트" 만 라우팅

9. 안티패턴 금지 목록

금지 1. Event Bridge 에 비즈니스 로직 넣기
