1 설계 배경 및 전략

왜 open-RMF를 사용하지 않는가?

항목open-RMFCasting-FMS
대상범용 (물류센터, 병원 등)주조 공정 특화
로봇 유형AMR 위주AMR + 로봇팔 (Cobot) 혼합
공정 모델단순 이동 (A → B)8단계 생산 공정 + 순서 제약
복잡도ROS2 + rmf_core 대규모필요한 모듈만 경량 구현
의존성ROS2 Galactic/Humble 강결합ROS2 Jazzy + 독립 Python

Casting-FMS 핵심 원칙

  • Just Enough: open-RMF의 ~40개 패키지 중 필요한 개념만 추출
  • 공정 순서 강제: 동일 item_id의 Task는 반드시 순서 보장
  • 이종 로봇 통합: AMR (이동) + Cobot (조작)을 하나의 Task 체인으로 관리
  • PostgreSQL 중심: 모든 상태를 DB에 기록 (PyQt5 + Web 양쪽 조회)
  • 점진적 확장: 3대 AMR로 시작, 구조 변경 없이 스케일 가능

레이어별 책임

Layer책임핵심 질문
Planning 무엇을, 누가 할 것인가 work_order → Task 분해, 최적 로봇 배정
Coordination 어디로, 언제 갈 것인가 경로 충돌 방지, 스테이션 점유 관리
Execution 실제로 움직여라, 잘 되고 있는가 ROS2 action 호출, 타임아웃/재시도
Data 지금 상태가 어떤가 로봇 위치/배터리, 공장 맵 그래프

3 보유 컴포넌트 (3개)

Planning 보유
Task Manager
work_order를 item_id별 Task로 분해. 생산 Task (5단계) + 적재 Task (2단계) + 출고 Task (3단계) 순서 보장.
입력: work_order → 출력: Task 배열 (순서 제약 포함)
Planning 보유
Task Allocator
가용 로봇 목록과 Task 조건을 받아 최적 로봇 선택. 거리, 우선순위, 배터리, capability 종합 스코어링.
입력: Task + 로봇 목록 → 출력: 특정 로봇에 명령
Coordination 보유
Traffic Manager
AMR 간 경로 충돌 방지. Waypoint 노드 잠금, Backtrack Yield 알고리즘으로 데드락 해소.
입력: 로봇 위치 + 경로 → 출력: 이동 허가 / 대기 신호
Confluence 정의 현황: 3개 모두 역할 정의는 완료. 기능 정의, 인터페이스 정의, 기술 스택 후보는 아직 미정.

4 추가 필수 컴포넌트 (4개)

Coordination 미보유 (필수)
Resource Reservation Manager
공장 내 정적 자원(스테이션, 버퍼, 독, 충전소)의 점유 상태 관리. Traffic Manager가 "이동 경로"를 관리한다면 이 매니저는 "머무르는 공간"을 관리.
관리 자원예시
Production Station주형 제작대, 주조 구역, 후처리 스테이션
Buffer Zone냉각 대기 공간, 검사 대기열
Dock적재 독, 출고 독
Charging StationAMR 충전소
없으면? Task Allocator가 로봇을 배정해도, 도착할 스테이션이 이미 점유 중이면 로봇이 앞에서 무한 대기하거나 충돌 발생.
Execution 미보유 (필수)
Robot / Cell Executor
Planning/Coordination 레이어의 결정을 실제 하드웨어 명령으로 변환하는 계층. 3개 매니저가 "누가, 어디로, 언제"를 결정해도, 이것 없이는 로봇이 움직이지 않음.
대상명령 인터페이스
AMR (RPi4)ROS2 nav2 Action Client (NavigateToPose)
Cobot 로봇팔 (RPi5)ROS2 MoveIt Action / 직접 제어
컨베이어 (ESP32)Serial JSON 명령 (9600 UART)
비전 검사ROS2 Service Call
없으면? 계획만 세우고 실행이 불가능. 두뇌는 있으나 팔다리가 없는 상태.
Execution 미보유 (필수)
Execution Monitor
Task 실행 중 이상 상태 감지 + 자동 복구를 담당하는 감시자. 로봇이 멈추거나 Task가 예상 시간을 초과해도, 이것 없이는 시스템이 인지 불가.
감시 항목대응 정책
Task 타임아웃5분 초과 시 경보, 10분 초과 시 Task 취소 + 재배정
로봇 응답 없음heartbeat 3회 누락 시 해당 로봇 비활성화
Task 실패최대 3회 재시도, 초과 시 관리자 알림
데드락 감지순환 대기 탐지 → Traffic Manager에 Backtrack 요청
없으면? 로봇 하나가 멈추면 전체 공장 라인이 정지. 무인 운영 불가능.
Data 미보유 (필수)
Fleet State Manager
모든 AMR/Cobot의 실시간 상태를 집중 관리하는 중앙 데이터 허브. Task Allocator, Traffic Manager, Execution Monitor 모두 이 데이터에 의존.
관리 데이터소스
위치 (x, y, theta)AMR ROS2 /odom 토픽
배터리 잔량 (%)AMR BMS 보고
상태idle / busy / charging / error / offline
현재 Task IDExecutor 보고
heartbeat1초 주기, 3회 누락 시 offline 전환
없으면? Task Allocator가 로봇을 배정하려 해도 "어떤 로봇이 어디에 있고, 배터리가 얼마인지" 알 수 없음.

5 권장 컴포넌트 (2개)

Data 권장
Map / Graph Server
공장 바닥의 Waypoint-Edge 그래프를 제공. Traffic Manager가 경로 충돌을 판단하려면 이 그래프가 필수. 노드 잠금, 최단 경로 계산, 동적 장애물 반영의 기반 데이터.
대안: Traffic Manager 내부에 하드코딩할 수 있지만, 공장 레이아웃 변경 시 코드 수정이 필요해 분리 권장. 현재 docs/fleet_traffic_management.html에 1m×2m 맵 + 0.12m AMR 3대 설계 완료.
Planning 권장
Battery / Charging Manager
AMR 배터리 상태를 모니터링하고 자동으로 충전 Task를 삽입. Task Allocator 내부에 넣을 수도 있지만, 충전 스케줄링 로직이 복잡해지면 분리가 유리.
배터리 레벨상태동작
> 30%정상일반 Task 수행
20~30%충전 필요현재 Task 완료 후 충전소 복귀 Task 자동 삽입
< 20%위험모든 새 Task 거부, 즉시 충전소 이동

8 갭 분석 요약

# 컴포넌트 Layer 상태 우선순위 미구현 시 영향
1 Task Manager Planning 보유 - -
2 Task Allocator Planning 보유 - -
3 Traffic Manager Coordination 보유 - -
4 Fleet State Manager Data 보유 (보강) - DB equipment 테이블로 이미 구현. heartbeat 수신 + offline 자동 전환 로직만 추가 필요
5 Robot / Cell Executor Execution 필수 P0 (최우선) 실행 레이어 부재 → 계획만 세우고 로봇이 움직이지 않음
6 Resource Reservation Coordination 필수 P1 스테이션 충돌 → 2대 이상 동시 도착 시 물리적 사고 위험
7 Execution Monitor Execution 필수 P1 장애 감지 불가 → 로봇 1대 정지 시 전체 라인 교착
8 Map / Graph Server Data 권장 P2 Traffic Manager 내부 하드코딩으로 대체 가능 (확장성 제한)
9 Battery / Charging Manager Planning 권장 P2 Task Allocator에 통합 가능 (충전 로직 복잡도 증가 시 분리)
구현 순서 제안: P0 (Robot Executor) → P1 (Resource Reservation + Execution Monitor) → P2 (Map Server + Battery Manager). Fleet State Manager는 기존 equipment 테이블 + heartbeat API 보강으로 대체. P0 없이는 보유한 3개 컴포넌트도 실제 동작 불가.
