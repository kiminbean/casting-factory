Monitoring (PyQt) — 진행 TASK 정리

monitoring/ · Factory Operator PC 관점 · 2026-04-20
본 문서는 backend/management/ 가 아니라 PyQt 관제 앱(monitoring/) 관점에서 아직 진행해야 할 TASK 만 모은 것이다. V6 결정(PyQt → gRPC 직결)이 적용 중이지만 api_client(HTTP + mock) 와 ws_worker(WebSocket) 가 여전히 공존한다. 대부분의 TASK 는 기존 HTTP/Mock 경로의 gRPC 이관, 실시간 채널 안정화, 운영성/배포 세 축으로 나뉜다.

P1 — 다음 스프린트 내

2. P0 — 가동 차단 요소

M-P0-01 P0 operations

패턴 등록 → 생산 시작 E2E 실기 검증

operations.py 의 [패턴 등록] → [생산 시작] 버튼은 POST /api/production/patterns 와 ManagementClient.start_production() 2단 호출. 두 호출 사이 실패 시 롤백 경로 없음 — 패턴은 등록됐는데 gRPC 가 터지면 UI 가 stuck 됨.
완료 조건승인 주문 1건 → 패턴 등록 → gRPC start_production 성공 + 실패 시 rollback UX (상태 라벨 에러 표시) 차단Mgmt 가 running 이어야 — 시연 직전 smoke test 필수
M-P0-02 P0 production

응급정지 / 운전모드 전환 버튼 TODO 해소

production.py:433,437 의 # TODO: POST /api/production/emergency_stop 와 # TODO: POST /api/production/mode 2건이 그대로 남음. 관제실에서 가장 자주 누르는 버튼이라 데모에서 실패하면 신뢰도 타격.
완료 조건Mgmt gRPC 에 해당 RPC 가 있으면 그쪽 호출로, 없으면 FastAPI endpoint 확인 후 1건씩 바인딩 선택지(A) FastAPI 에 이미 있는지 확인 · (B) Mgmt 에 RPC 추가 제안
M-P0-03 P0 workers

gRPC stream 재연결/keepalive 실전 검증

4개 스트림 워커(items/alerts/camera/amr_status) 가 Mgmt 프로세스 재시작/TCP 끊김 상황에서 자동 복구하는지 확인. ManagementClient keepalive 옵션은 있으나 워커 레벨 backoff 루프가 일관적이지 않음.
완료 조건서버 재시작 3회 → 워커 전부 30초 내 재구독 · 로그에 "reconnect" 기록 테스트Mgmt 를 SIGTERM 후 재기동, camera_frame_worker 에 last_seq 유지 여부 확인
M-P0-04 P0 critical alert

Critical 모달 폭주 캐시 cap 교체

main_window._on_alert_event 의 cap(200→100) 방식은 순서 보장 없음 — set(list(...)[-100:]) 은 iteration order 에 의존. 대량 critical 시 실제로는 최근 alert_id 가 날아갈 수 있음. deque 또는 OrderedDict 로 교체.
완료 조건LRU 보장 + 단위 테스트 (300건 주입 → 최신 100건만 남는지)

3. P1 — 다음 스프린트

M-P1-01 P1 migration

HTTP → gRPC 이관 (Phase 8 완료)

api_client.py 400+ 줄 중 대다수가 mock fallback 에 의존. 다음 순서로 Mgmt gRPC 전환:
① dashboard KPI (/api/dashboard/stats) → Mgmt 집계 RPC 신설 요청
② logistics tasks (/api/logistics/tasks) → ListItems(stage=TR_*) 파생
③ quality summary (/api/quality/summary) → Mgmt 또는 DB 직결 RPC
완료 조건3개 엔드포인트에 대해 CASTING_DATA_MODE=normal 에서도 mock 없이 화면 동작 주의RPC 없는 것 먼저 백엔드팀과 합의 (맘대로 proto 추가 금지)
M-P1-02 P1 handoff

Handoff ACK 수신 경로 gRPC 로 전환

현재 _on_handoff_ack 는 WebSocket handoff.ack 만 수신. WebSocket 단종 기본값(CASTING_WS_ENABLED=0) 환경에서 Factory Operator 가 인수인계 사실을 알지 못함. WatchAlerts 에 handoff 이벤트를 얹거나 WatchHandoff RPC 신설.
완료 조건WS 없이 handoff 발생 → 8초 상태바 메시지 + 로그 기록 연관SPEC-AMR-001 FR-AMR-01-06
M-P1-03 P1 logistics

AMR 상태 폴링 → stream 전환

AmrStatusWorker 가 AMR_POLL_INTERVAL=10초 폴링. Mgmt 에 WatchRobotStatus stream 이 있으면 10초 지연 제거. 현재 스트림이 없으면 프로토 확장 요청, 폴링 유지 시 간격을 배터리/상태별 적응형으로 (IDLE 30s / MOVE 2s).
완료 조건MOVE 상태에서 UI 마커가 2초 내 갱신 (현재 10초)
M-P1-04 P1 amr card

수동 상태 전이 버튼 (UNLOADING / UNLOAD_COMPLETED / RESET)

transition_amr_state() 는 준비돼 있으나 UI 진입점 없음. AmrStatusCard 또는 logistics 페이지에 "하차 시작 / 완료 / FAILED 리셋" 버튼 3종 추가. Factory Operator 가 ROS 단절 시 수동 복구 가능해야 함.
완료 조건3개 버튼 → AmrStateMachine 10-state 전이표 준수 + 실패 시 QMessageBox 사유 표시
M-P1-05 P1 mqtt

MQTT 경로 정리 (사용처 vs Mgmt 중복)

mqtt_worker 가 conveyor/+/status, vision/+/result, cobot/+/status, amr/+/position, factory/alarm 구독. Mgmt gRPC stream 과 일부 중복 가능성. 표를 만들어 어느 채널이 source of truth 인지 1건씩 확정 후 중복 제거.
완료 조건5개 토픽 각각 (A) MQTT keep / (B) gRPC 전환 / (C) 제거 결정된 표
M-P1-06 P1 refresh

REFRESH_INTERVAL 전역 8초 → 페이지별 차등

REFRESH_INTERVAL_MS=8000 이 모든 페이지에 동일 적용 중. dashboard KPI 는 8초여도 되지만 production 진행 테이블은 2~3초가 필요. map / dashboard 미니맵만 500ms fast timer 가 별도로 돈다 — 일관성 없음.
완료 조건페이지별 interval config + stream 도달 시 interval 자동 slowdown

4. P2 — 배포/안정화 전

M-P2-01 P2 data mode

mock_data 의존 축소 + 데모 모드 명시

CASTING_DATA_MODE 3-level(normal/fallback/mock_only) 이 있으나 페이지들은 fallback 을 암묵 전제로 짜여있음. 실제 운영에서 mock 이 보이면 심각한 혼동. 상태바에 현재 모드 배지 노출 + mock_only 는 "DEMO" 워터마크.
완료 조건우하단 "MODE: normal/fallback/DEMO" 배지 + 데모 모드 배경 워터마크
M-P2-02 P2 tls

mTLS 클라이언트 cert 자동 배포

MGMT_GRPC_TLS_ENABLED=1 활성 시 backend/management/certs/{ca,client}.{crt,key} 를 찾음. Operator PC 가 여러 대면 cert 배포 수작업 부담. monitoring/scripts/fetch_certs.sh (SCP 또는 Tailscale) 추가.
완료 조건1 명령으로 cert 3종 다운로드 + 만료일 체크 경고
M-P2-03 P2 env

환경변수 네임스페이스 일원화

현재 혼재: CASTING_API_HOST, CASTING_MQTT_HOST, MANAGEMENT_GRPC_HOST, MGMT_GRPC_TLS_ENABLED, MGMT_IP_CAMERA_ID. 접두사를 CASTING_ 으로 통일하고 MGMT_ 를 alias 로만 남기는 이행 계획.
완료 조건.env.example 에 모든 변수 정리 + 배포 문서 업데이트
M-P2-04 P2 packaging

PyInstaller 패키징 + 관제실 고정 레이아웃

README roadmap v1.0.0 항목. macOS(dev) + Ubuntu 24.04(현장) 양쪽 빌드. 기동 시 자동 전체화면, ESC 차단 옵션(CASTING_KIOSK=1).
완료 조건.app / .AppImage 생성 + 의존 3rd party (grpc, pyqt5, requests) 번들
M-P2-05 P2 logging

로그 파일 rotation + 에러 집계

현재 logging.basicConfig 만 사용 — 파일 저장 없음. 관제실 PC 에서 장시간 구동 후 사고 분석 불가. RotatingFileHandler + critical 레벨 이상은 별도 파일로.
완료 조건~/monitoring-logs/ 에 10MB x 5 rotation + critical.log 분리
M-P2-06 P2 tests

pytest 도입 — 최소 smoke

monitoring/ 쪽에 테스트 0건. 우선 ManagementClient 단위 테스트(mock stub) + operations._on_register_pattern 워크플로 smoke 3개만. GUI 테스트는 비용 대비 낮은 우선순위.
완료 조건pytest monitoring/ 에서 녹색 5건 이상 + CI 주입
